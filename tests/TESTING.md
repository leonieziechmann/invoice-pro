# Testing Guide

This project uses [**tytanic**](https://github.com/tingerrr/tytanic) as its test runner for visual regression and data-level integration tests.

## Directory Structure

Every test case lives in its own directory under `tests/` and **must** contain at least a `test.typ` file. Tytanic generates additional folders (`out/`, `diff/`) during runs — these are git-ignored.

```
tests/
├── data-test.typ          # Shared utility: data-level assertions
├── test-locale.typ        # Shared utility: stable locale for tests
│
├── docs/                  # Tests that mirror documentation examples
│   ├── getting-started-minimal/
│   │   ├── .gitignore
│   │   ├── test.typ
│   │   └── ref/           # Reference snapshots (committed)
│   └── …
│
├── integration/           # End-to-end rendering tests
│   ├── features-complex/
│   │   ├── .gitignore
│   │   ├── test.typ
│   │   └── ref/
│   └── features-sparse/
│       └── …
│
└── line-items/            # Focused unit-style tests for calculations
    └── totals/
        ├── .gitignore
        └── test.typ       # Uses data-test for value assertions
```

### Naming Conventions

| Level                                                 | Purpose                     | Examples                                                                                         |
| ----------------------------------------------------- | --------------------------- | ------------------------------------------------------------------------------------------------ |
| **Category** (`docs/`, `integration/`, `line-items/`) | Groups tests by intent      | `docs/` = documentation parity, `integration/` = full renders, `line-items/` = calculation logic |
| **Test case directory**                               | Descriptive kebab-case name | `features-complex`, `totals`, `getting-started-minimal`                                          |
| **Test file**                                         | Always `test.typ`           | Required by tytanic                                                                              |

### The `.gitignore` Requirement

Every test case directory **must** contain a `.gitignore` file with the following content:

```gitignore
# generated by tytanic, do not edit

/diff/
/out/
```

This prevents tytanic's generated output (`out/` — rendered pages, `diff/` — visual diffs) from being committed. The `ref/` directory, which holds reference snapshots, **is** committed.

> **Tip:** When creating a new test, copy an existing `.gitignore` or use tytanic's tooling to generate it automatically.

## Shared Utilities

Two helper files in the `tests/` root are available for import in any test.

### `test-locale` — Stable Locale

**File:** `tests/test-locale.typ`

Standard locales (e.g., `locale.de-de`) may change between releases. To keep tests deterministic, use `test-locale` instead:

```typ
#import "/tests/test-locale.typ": test-locale

#show: invoice.with(
  locale: test-locale,
  // …
)
```

`test-locale` provides:

- A **stable language** derived from `base-language` (no translated labels that might shift).
- A **stable region** derived from `base-region` with:
  - A fixed currency: `EUR` (`€`, 2 decimals, 4 fine decimals).
  - A deterministic tax inference: all tax rates resolve to `tax.vat(rate)`.

**When to use:** Any test that asserts on computed values (totals, tax breakdowns) or rendered text should use `test-locale` to avoid false failures from upstream locale changes.

**When to skip:** Integration or documentation tests that intentionally verify a real locale (e.g., `locale.en-de`) can use it directly — but accept that those tests may need updating when the locale definition changes.

---

### `data-test` — Value-Level Assertions

**File:** `tests/data-test.typ`

Visual regression tests catch rendering changes but not logical errors in calculations. `data-test` lets you assert on the **computed data** inside the loom tree without relying on pixel output.

#### How It Works

`data-test` is a loom `motif` that hooks into the draw phase. It receives the loom context (`ctx`) and the aggregated data from its children (`children-data`), then runs a user-supplied test function:

```typ
#import "/tests/data-test.typ": data-test, loom

#data-test(test: (ctx, data) => {
  // `data` contains computed signals from children
  // `ctx` is the full loom context
  // Use assert.eq() to validate values
})[
  // … child components whose data you want to inspect
]
```

#### Example: Asserting Line-Item Totals

```typ
#import "/src/lib.typ": *
#import "/tests/data-test.typ": data-test, loom
#import "/tests/test-locale.typ": test-locale

#show: invoice.with(
  theme: themes.blank,
  locale: test-locale,
  sender: (name: "Test Sender", address: "Street 1", city: "City"),
  recipient: (name: "Test Recipient", address: "Street 2", city: "City"),
)

#data-test(test: (ctx, data) => {
  // Extract the line-items signal from the children data
  let line-items-data = loom.query.find-signal(data, "line-items")

  assert.eq(
    line-items-data.total.net,
    decimal("100"),
    message: "Net total: expected 100.00, got " + repr(line-items-data.total.net),
  )
  assert.eq(
    line-items-data.total.gross,
    decimal("119"),
    message: "Gross total: expected 119.00, got " + repr(line-items-data.total.gross),
  )
})[
  #line-items[
    #item(
      [Test Item],
      price: 100.00,
      quantity: 1,
      tax: tax.vat(19%),
    )
  ]
]
```

Key details:

- **`loom.query.find-signal(data, "line-items")`** extracts the signal published by the `#line-items` component, which contains the full totals breakdown.
- **`decimal("100")`** — use `decimal()` for exact numeric comparisons (avoids floating-point mismatches).
- **`themes.blank`** — use the blank theme when you only care about data, not visual output.
- **Tax constructors** — use `tax.vat(rate)` for the standard rate, `tax.lower-rate(rate)` for reduced rates, and `tax.zero()` for zero-rated items. Each produces a different tax category.
- **Assertion messages** — always include both the expected and actual value in the message for fast debugging. Use the pattern: `"Field: expected <value>, got " + repr(actual)`. The `repr()` function ensures the actual value is displayed in a readable format.
- **Valid IBAN/BIC** — when using `bank-details` in tests or docs, always use values that pass validation checks. Use IBAN `DE75512108001245126199` and a valid 9 or 11 character BIC (e.g., `SOLADEST600`). Fake values like `DE12 3456 7890...` or `EXAMPLEBICX` will fail IBAN/BIC validation.
- Always combine `data-test` with `test-locale` for deterministic results.

## Writing a New Test

### 1. Visual Regression Test (Integration / Docs)

These tests render a full invoice and compare it pixel-by-pixel against a reference snapshot.

```
tests/<category>/<test-name>/
├── .gitignore       # Required — see above
├── test.typ         # The test source
└── ref/             # Must be created then can be update with `tt update`
    └── 1.png
```

**Steps:**

1. Create the directory and add the `.gitignore`.
2. Write `test.typ` — a self-contained Typst document that renders the scenario you want to test.
3. **Important:** You must manually create the `ref/` directory and an empty `1.png` file inside it (e.g. `ref/1.png`). If this file doesn't exist, tytanic will treat it as a compile-only test rather than a visual regression test.
4. Run `tt update <test-path>` to generate the actual reference snapshot over the empty file.
5. Commit `test.typ`, `.gitignore`, and the `ref/` directory.

```typ
// tests/integration/my-new-test/test.typ
#import "/src/lib.typ": *

#show: invoice.with(
  theme: themes.DIN-5008(font: "libertinus serif"),
  sender: (name: "Test Corp", address: "1 Test St", city: "12345 Testville"),
  recipient: (name: "Client", address: "2 Client Ave", city: "54321 Clientown"),
  invoice-nr: "TEST-001",
)

#line-items[
  #item(
    [Service],
    price: 100.00,
    quantity: 2,
    modifier: discount([Special Offer], amount: 10%),
  )
]
```

### 2. Data Assertion Test (Line-Items / Calculations)

These tests validate computed values without depending on rendering output.

```
tests/line-items/<test-name>/
├── .gitignore
└── test.typ         # Uses data-test + test-locale
```

**Steps:**

1. Create the directory and add the `.gitignore`.
2. Write `test.typ` using `data-test` and `test-locale` (see example above).
3. **Important:** For unit-like data tests, you should use `themes.blank` as the theme. Note that `themes.blank` is a value, not a function, so use it directly: `invoice.with(theme: themes.blank)`. This ensures that the minimal possible amount of code is executed to narrow the test scope.
4. Run `tt run` — the test passes if all `assert.eq()` calls succeed.

> **Note:** Data tests may still produce visual output (and thus `out/`), but the assertions are what determine pass/fail.

---

### 3. Documentation Example Tests (`docs/`)

Every non-trivial code block in the documentation (`docs/docs/**/*.md`) must have a corresponding test under `tests/docs/`. These tests ensure that published examples remain compilable and visually correct as the codebase evolves.

```
tests/docs/<doc-page>-<example-name>/
├── .gitignore
├── test.typ
└── ref/
    └── 1.png
```

**Steps:**

1. Identify each non-trivial code block in the documentation page.
2. Create a test directory under `tests/docs/` with a descriptive kebab-case name derived from the doc page and example (e.g., `getting-started-minimal`).
3. Copy the code block into `test.typ`, adjusting imports to use `/src/lib.typ` instead of the package import `@preview/invoice-pro:...`.
4. Add the `.gitignore`, create `ref/1.png`, run `tt update`, and commit.

**Resolving discrepancies between docs and tests:**

When the documentation code and the test code diverge, use the following rule of thumb:

> **Take syntax from the test, take structure from the docs.**

- **The docs are the target** — if the docs show a different scenario, feature usage, or overall structure, update the test to match the docs.
- **The tests are likely correct for syntax** — if the difference is small (renamed parameters, updated function signatures, changed API surface), the test has probably been updated to match a code change that the docs haven't caught up with yet. In this case, update the docs.
- **Use context to decide** — if a parameter was renamed in the source but the docs still use the old name, the test is correct. If the docs intentionally demonstrate a new pattern, the docs are correct. Infer intent from the surrounding changes.

## Running Tests

```bash
# Run all tests
tt run

# Run a specific test
tt run "line-items/totals"

# Update reference snapshots after intentional visual changes
tt update

# Update a specific test's reference
tt update "integration/features-complex"
```

## Quick Checklist for New Tests

- [ ] Created a new directory under the appropriate category
- [ ] Added a `test.typ` file
- [ ] Added the `.gitignore` (with `/diff/` and `/out/`)
- [ ] Used `test-locale` if asserting on computed values or rendered text
- [ ] Used `data-test` if validating calculations (not just visual output)
- [ ] Generated reference snapshots with `tt update` (for visual tests)
- [ ] Committed `test.typ`, `.gitignore`, and `ref/` (if applicable)
